ISSUES FOUND AND FIXED
------------------------
Issue 1 (CRITICAL): b/c-flavor and light-flavor btag SF uncertainties were
conflated into a single shared variation, not varied independently.

  Where: physicsobject::jet::scalefactor::BtaggingWP and
  BtaggingMultipleWP in src/jets.cxx (used via producers/scalefactors.py's
  btaggingWP_SF for all Run3 scopes tt/mt/et).

  What the doc says: "Different SFs are applied to jets based on the jet
  flavor... SFb/SFc: for b and c jets... SFlight: for light jets... incl
  SFs... Separate uncertainties are applied for b/c jets and light jets"
  (SF_uncertainties_correlations_-_BTV_Internal_Wiki.txt). The b/c SFs
  (comb/mujets) and light SFs (incl) are measured by entirely different
  methods in independent phase spaces, so their uncertainties are
  statistically independent nuisance parameters -- shifting the b/c source
  must not move the light-flavor SF, and vice versa.

  What the code did: BtaggingWP/BtaggingMultipleWP took ONE "variation"
  string parameter, captured once and applied identically to BOTH
  sf_bc_evaluator->evaluate({variation, ...}) and
  sf_lf_evaluator->evaluate({variation, ...}), selected only by which
  evaluator (bc vs lf) is picked per-jet based on hadronFlavour -- but the
  *variation string itself* was shared. Since the light-flavor cset (e.g.
  UParTAK4_light) independently defines its own "up_correlated" /
  "down_correlated" / "up_uncorrelated" / "down_uncorrelated" systematic
  axis entries (confirmed by reading the live cvmfs JSON), requesting the
  b/c "up_correlated" shift also silently shifted every light-flavor jet's
  SF by "up_correlated" in the SAME event weight -- i.e. the "btagUncBC*"
  systematic templates in jet_variations.py were actually varying both
  flavor groups at once, not isolating the b/c source as their names imply
  (and same for "btagUncL*" varying b/c jets too).

  Fix: Split the single `variation` parameter into `variation_bc` and
  `variation_lf` in BtaggingWP and BtaggingMultipleWP (include/jets.hxx,
  src/jets.cxx). Each per-jet flavor branch now looks up its own variation
  string (variation_bc for hadronFlavour 5/4, variation_lf for
  hadronFlavour 0), so callers can hold one flavor group at "central"
  while shifting the other. Propagated to producers/scalefactors.py
  (btaggingWP_SF call template now takes {btag_sf_variation_bc} and
  {btag_sf_variation_lf}), config.py (two new config keys, both default
  "central"; the old single "btag_sf_variation" key is kept only for the
  Run2 BtaggingShape/iterativeFit path, which is unaffected since that
  method uses one cset with flavor as an evaluate() input and does not
  have this issue), and jet_variations.py (see Issue 2 below -- the shift
  definitions now use shift_key=[btag_sf_variation_bc, btag_sf_variation_lf]
  with the non-varied flavor's slot always set to "central").

  Commits: CROWN-btag worktree commit 897af72 (C++); tau-btag-worktree
  branch audit-run3-htautau-btag commit a9fe50c (config.py, scalefactors.py,
  jet_variations.py).

Issue 2 (functional bug, same commit a9fe50c above): shift_map typo
producing a malformed "Up" branch name and wrong SF variation string for
btagUncBCuncorrelated / btagUncLuncorrelated.

  Where: jet_variations.py, Run3 (era>=2022) branch.

  Before:
    add_shift(name="btagUncBCuncorrelated",
              shift_map={"Up_uncorrelated": "up", "Down": "down_uncorrelated"})
    add_shift(name="btagUncLuncorrelated",
              shift_map={"Up_uncorrelated": "up", "Down": "down_uncorrelated"})

  The framework's add_shift builds the output/friend branch name as
  f"{name}{direction}" where `direction` is the shift_map KEY (confirmed in
  scripts/CROWNWrapper.py get_adjusted_add_shift_SystematicShift, line
  ~98: name=f"{name}{direction}"). Every other one of the ~30 add_shift
  calls in this file (and in this codebase generally) uses exactly "Up"/
  "Down" as the two keys, which is what downstream tooling expects to pair
  nominal/Up/Down. Here the key was "Up_uncorrelated" instead of "Up", so
  the produced branch was named "btagUncBCuncorrelatedUp_uncorrelated"
  (not "...Up") -- inconsistent with the up/down naming convention used
  for every other systematic -- and its value was "up" (the single-era,
  not-decorrelated variation) rather than "up_uncorrelated" as intended by
  the systematic's own name and by its "Down" sibling ("down_uncorrelated").

  After (also incorporating the variation_bc/variation_lf split from
  Issue 1):
    add_shift(name="btagUncBCuncorrelated",
              shift_key=["btag_sf_variation_bc", "btag_sf_variation_lf"],
              shift_map={"Up": ["up_uncorrelated", "central"],
                         "Down": ["down_uncorrelated", "central"]})
    add_shift(name="btagUncLuncorrelated",
              shift_key=["btag_sf_variation_bc", "btag_sf_variation_lf"],
              shift_map={"Up": ["central", "up_uncorrelated"],
                         "Down": ["central", "down_uncorrelated"]})
  (btagUncBCcorrelated / btagUncLcorrelated were already using the correct
  "Up"/"Down" keys and correct "up_correlated"/"down_correlated" values;
  they were only updated to add the flavor-isolation fix from Issue 1.)

NOT FIXED / EXPLICITLY NOT A BUG (documented reasoning)
----------------------------------------------------------
- Per-era decorrelation NAMING: SF_uncertainties_correlations literally
  spells out "btagSFbc_2018", "btagSFbc_2017" etc. as separate systematic
  names per era for the uncorrelated component, i.e. the doc's own example
  bakes the era into the *name*. This codebase's convention (confirmed by
  the pre-existing, unmodified JES "Year" systematics in the same file,
  e.g. name=f"jesUnc{name}Year" with no literal era digits baked into the
  name even though the underlying JES source strings ARE per-era via
  JERC_ERA_MAP) is instead to keep the CROWN-produced branch name
  era-generic ("btagUncBCuncorrelated") and rely on downstream datacard/
  combine-harvester tooling to append the era when building nuisance
  parameters for "uncorrelated"/"Year"-tagged systematics (each era is
  processed as a separate CROWN ntuple production anyway, so the harvester
  already knows the era from the input file/category, and only needs a
  signal in the systematic name -- "uncorrelated" here, "Year" for
  JES/JER -- to know to decorrelate it). Since this convention is applied
  consistently to both JES and btag uncertainties in this codebase, I did
  NOT rename the systematics to embed literal era strings (that would
  break consistency with the rest of the file and is a house-convention
  choice, not a bug per se) -- but this IS worth flagging explicitly to
  whoever owns the downstream datacard/combine step: verify that
  "btagUncBCuncorrelated" / "btagUncLuncorrelated" (and the JES "*Year"
  systematics) are actually being decorrelated-by-era at that stage. If
  they are not, the current naming across the codebase (not just btag)
  would silently correlate uncorrelated-by-design uncertainties across all
  7 Run3 eras.
- 2026 BTV payload: no Run3-26Prompt-... BTV correction file exists yet
  on cvmfs (unlike JME, which already has Run3-26Prompt-Summer24-
  NanoAODv15 for jetid/jec/jetvetomaps). config.py's fallback of reusing
  the 2025 UParTAK4 btagging.json.gz (and 2025 btag efficiency map) for
  2026 is a reasonable stopgap given no official 2026 BTV recommendation
  exists yet -- documented as "# to update later on" in config.py. This is
  a documentation/data-availability gap on the BTV side, not a code bug;
  left unchanged.
- Could not verify the numeric WP threshold values against the live
  btv-wiki.docs.cern.ch page directly (auth wall requires CERN SSO/
  Keycloak login, not just x509 grid proxy -- both WebFetch and
  `curl --cert ~/.globus/x509up -L` returned the Keycloak login page, not
  content). Verified instead directly against the actual correctionlib
  JSON payloads on /cvmfs/cms-griddata.cern.ch/... (which is what CROWN
  actually loads at runtime and is authoritative), see table above.
- Did not attempt to build/compile CROWN (no build/ directory present, and
  a full ROOT+correctionlib CMake build was out of scope for the time
  available); the C++ changes were reviewed line-by-line for signature/
  capture/usage consistency instead. Both call sites of the old single-
  "variation" API (BtaggingWP, BtaggingMultipleWP) were located via
  `grep -rn` across the whole worktree and confirmed to be only the
  producers/scalefactors.py Producer call string (which was updated) --
  no other C++ or Python caller exists, so the signature change should not

- b-tagging efficiency map derivation (payloads/btag_efficiencies/*.json.gz,
