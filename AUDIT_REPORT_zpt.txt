===========================================================================
1. Kinematic variable and DY-only gating — CORRECT, no change
===========================================================================
Doc: "Input 3: ptll — Gen-level pTll: Obtained from gen-level Electrons/Muons
with status==1, or gen-level Taus with status==2, both requiring status flag
'fromHardProcess'." The correctionlib example snippet sets `ptll = FullGenPt`
(the *full* gen boson pT, not the visible one).

Code: event.py ZPtReweighting producer (lines 95-99) takes q.genboson_p4 as
input, which is built by genparticles::GetBoson (src/genparticles.cxx:90-153).
GetBoson sums four-vectors of status==1 leptons with the fromHardProcessFinalState
flag (bit 8) OR particles with the isDirectHardProcessTauDecayProduct flag
(bit 10) — including neutrinos. This reproduces the tau's full four-momentum
(equivalent to using the tau itself at status==2, as the doc describes) and
therefore yields the "Full genlevel V_pt" required by the correction, matching
the doc's example (`ptll = FullGenPt`). This is distinct from
genparticles::GetVisibleBoson (visgenboson_p4), which explicitly drops
neutrinos and is correctly NOT used here.

Gating to DY MC only: config.py modification rules (line ~1532-1538) apply
event.ZPtReweighting only to
samples=["dyjets_powheg","dyjets_amcatnlo","dyjets_amcatnlo_ll",
"dyjets_amcatnlo_tt","electroweak_boson"] for era >= 2022, i.e. DY (and the
combined single-EWK-boson) MC only — never data/embedding, never other
backgrounds. Correct and consistent with the doc's intent ("Weights that are
to be applied on DY MC samples").
The C++ ZBosonPt() for real data isn't even reached since GetBoson returns a
default (zero) vector for is_data==true and the producer is never appended
for data samples in config.py — no risk of a bogus non-unity weight on data.

3. Systematic uncertainty — WRONG, FIXED (main finding)
===========================================================================
Doc: "The number of uncertainties depends on the order of the used DY
samples. It's 8 for LO samples, 10 for NLO samples and 9 for NNLO samples...
Each uncertainty corresponds to the uncertainty of one fit parameter...
Therefore, all uncertainties are required to be included and are fully
uncorrelated from each other." The correctionlib "syst" input accepts
'nom','up1','down1','up2','down2',... i.e. N independent nuisance parameters,
and DY_pTll_reweighting_N_uncertainty(order) gives N per order.

Code (before fix), config.py ~2470-2481:
    if int(era[:4]) >= 2022:
        add_shift(
            scopes=("et","mt","tt","em","ee","mm"),
            producers=[event.ZPtReweighting],
            shift_key="zpt_variation",
            name="zPtReweightWeight",
            shift_map={"Up":"up","Down":"down"},
            samples=["dyjets_powheg","dyjets_amcatnlo","dyjets_amcatnlo_ll",
                     "dyjets_amcatnlo_tt","electroweak_boson"],
        )
This registers exactly ONE systematic ("zPtReweightWeight") using the
variation strings "up"/"down". Looking at the underlying C++
(event::reweighting::ZBosonPt, src/reweighting.cxx:600-628), variation=="up"/
"down" triggers an *envelope* computation: it loops over all N up_i/down_i
variations and takes the maximum (for "up") / minimum (for "down") deviation
from nominal, collapsing all N independent fit-parameter uncertainties into a
single fully-correlated up/down pair. This directly contradicts the doc's "all
uncertainties are required to be included and are fully uncorrelated from each
other" — an envelope both loses information (only the single most extreme
parameter shift survives) and wrongly correlates parameters that must be
treated independently in the fit/limit-setting.

Fix applied (config.py ~2476-2510, committed): replaced the single envelope
shift with per-fit-parameter, fully-uncorrelated shifts named
"zPtReweightWeight_unc{n}" (Up -> "up{n}", Down -> "down{n}"), split by DY
sample order so each group only gets the number of parameters valid for its
order:
  - dyjets_powheg (NNLO): unc1..unc9 (9 uncertainties, per doc)
  - dyjets_amcatnlo / dyjets_amcatnlo_ll / dyjets_amcatnlo_tt /
    electroweak_boson (NLO): unc1..unc10 (10 uncertainties, per doc)
This follows this codebase's own established convention for decorrelated,
per-index systematics (e.g. the tau vsJet/vsMu ID SF shifts in config.py,
which loop over DM/pT/wheel bins and register one add_shift per index with
names like "vsMuWheel{wheel}" / shift_key "tau_id_vsmu_wheel{wheel}" — see
config.py ~2537-2591), so the naming style ("<Weight>_unc{n}") and mechanism
(multiple add_shift calls with disjoint `samples=` covering different DY
orders under the same shift-name index) are consistent with how other
decorrelated/binned CROWN systematics are already implemented in this file.
The underlying C++ (ZBosonPt) already supports "upN"/"downN" as literal
variation strings via its else-branch (src/reweighting.cxx:629-634,
`variation.rfind("up") != npos` -> `corrDY->evaluate({order, pt, variation})`),
so no C++ change was needed — only the config.py shift registration was wrong.
No change to the nominal event.ZPtReweighting producer/weight itself (only the
systematic-shift registration), so central-value yields are unaffected.

Note: the LO (madgraph) order (8 uncertainties) is defined in DY_PTLL_N_UNC
for completeness/documentation but LO samples (dyjets_madgraph / bare
"dyjets") are not applied to event.ZPtReweighting for Run3 (era>=2022) in this
config — Run3 DY samples in this analysis are amcatnlo/powheg only, and bare
"dyjets"/"electroweak_boson" for era<2022 instead use the legacy Run2
ZPtReweighting_Run2 workspace-based method, gated separately (config.py
~1944-1950, `if int(era[:4]) < 2022`). This pre-existing era separation is
correct and was not touched.

4. Cross-era correlation — NOT FIXED, flagged for downstream analysts
===========================================================================
The doc's changelog is explicit that each era has its own independently
fit correction file/parameters ("Two separate inputs are now required for the
era ... and for the order of the MC samples"), implying the per-parameter
uncertainties are almost certainly derived independently per era and should
not automatically be treated as correlated across eras (2022preEE vs
2022postEE vs 2023preBPix vs ... vs 2024). However:
- The doc's DY kinematic reweighting page does not explicitly state a
  cross-era correlation recommendation (unlike some other CMS systematics
  pages that spell out "correlated"/"uncorrelated" naming conventions), so I
  could not find an authoritative statement to act on with high confidence.
- This CROWN config (config.py) is generated once per era, and I found no
  existing convention anywhere in this file (searched for "correlated"/
  "Correlated" — zero hits) for era-suffixing shift names to force
  decorrelation; cross-era (de)correlation for ANY systematic in this analysis
  appears to be handled downstream (at the datacard/combine stage, by whether
  shift names match exactly across per-era CROWN outputs), not inside CROWN
  itself. Since "zPtReweightWeight_unc{n}" is named identically for every era,
  downstream combination will treat it as fully correlated across eras unless
  the combine/datacard tooling explicitly decorrelates it (e.g. via
  per-era categories or renaming).
I did NOT add era-suffixing to the shift names because doing so would deviate
from the repo's existing systematic-naming conventions (no other systematic in
this file is era-suffixed at the CROWN level) and I have no explicit doc
citation mandating it. This is flagged here as an open question for the
analysis owners / statistics team to resolve at the combine/datacard level,
not a CROWN code bug per se.
