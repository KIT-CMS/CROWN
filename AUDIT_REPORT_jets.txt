FINDINGS — FIXED
======================================================================

1) CRITICAL: unresolved git merge-conflict markers in include/jets.hxx
   File: include/jets.hxx (CROWN worktree)
   The header contained two literal `<<<<<<< HEAD / ======= / >>>>>>>`
   conflict blocks (leftover from merge commit c5e2cdd, "formatting"),
   in the PtCorrectionL2L3 and PtCorrectionMC declarations. This is a
   hard compile break — nothing in src/jets.cxx that includes this
   header would build. Both sides of the conflict were semantically
   identical (only line-wrapping differed), so I resolved by keeping
   the compact form and removing the markers.
   Commit: d4228ae "Fix unresolved merge-conflict markers in
   include/jets.hxx"

2) BUG: broken eta-range guard in the 2024 L2L3Residual low-pT
   workaround
   File: src/jets.cxx, PtCorrectionL2L3(), ~line 305
   Before:
     if (era == "2024" && pt < 30.0 && 2.0 < abs(eta) < 2.5)
         _pt = 30.0;
   In C++, `2.0 < abs(eta) < 2.5` parses as `(2.0 < abs(eta)) < 2.5`.
   The left comparison yields a bool (0 or 1), and both `0 < 2.5` and
   `1 < 2.5` are true — so the eta condition was a permanent no-op:
   the pt-floor override (used only for data, to keep the L2L3Residual
   evaluator's pT argument in-range in a problematic detector region
   for 2024) was silently applied for ALL eta, not just the intended
   2.0<|eta|<2.5 transition region. `git log -p` on this file shows the
   same broken pattern has existed since this workaround was first
   introduced, so it was never doing what the surrounding logic
   intended.
   Fixed to: `2.0 < std::abs(eta) && std::abs(eta) < 2.5`
   Commit: ce9acd7 "Fix broken eta-range guard in 2024 L2L3Residual
   low-pT workaround"
   This code path is the one actually used for Run3 data (PtCorrectionL2L3
   is shared between data and MC for era>=2022 — see config.py, no
   ReplaceProducer swap to a data-only variant for Run3), so the bug
   was live for 2024 data processing.

3) BUG: inverted REGROUPED flag → Run3 was using the full/individual
   JES uncertainty-source scheme instead of the JERC-recommended
   reduced ("Regrouped") scheme, and vice versa for Run2
   File: analysis_configurations/tau/jet_variations.py, add_jetVariations()
   Before:
     REGROUPED = True if int(era[:4])<2022 else False
   `git log -p` on this file shows two prior, separately-authored
   states of this exact line: `REGROUPED = True  # preferred
   configuration` (era-independent, from what was clearly a Run3-only
   version of the file) and `REGROUPED = False  # preferred
   configuration for run2` (from a Run2-only version). When the two
   were merged into a single era-conditional expression, the condition
   was flipped: the merged code set REGROUPED=True for era<2022 (Run2)
   and REGROUPED=False for era>=2022 (Run3) — exactly backwards from
   both prior "preferred configuration" comments.
   Effect of the bug: for all Run3 eras (2022preEE..2026), `if not
   JES_CONFIG.REGROUPED:` was true, so the code walked the ~19-source
   "full" individual JES uncertainty list (SinglePionECAL,
   AbsoluteMPFBias, RelativeJEREC1, RelativeSample, ...) with every
   source registered as `CMS_scale_j_{name}` with NO era suffix at
   all — i.e. every Run3 JES source was treated as fully correlated
   across 2022preEE..2026, including sources (AbsoluteStat, TimePtEta,
   RelativeStatFSR, RelativeJEREC1/2, RelativePtEC1/2, RelativeStatEC,
   RelativeStatHF, RelativeSample) that per the JERC "Uncertainty
   Sources" doc's own decomposition (time-dependence, relative
   eta-dependence categories) must be decorrelated between data-taking
   periods. Meanwhile Run2 (era<2022) was incorrectly running the
   "preferred configuration" reduced/regrouped scheme (Absolute,
   FlavorQCD, BBEC1, HF, EC2, RelativeBal, plus their yearly
   Absolute_2016/BBEC1_2016/... variants) that this code was actually
   written for Run3.
   Fixed to: `REGROUPED = True if int(era[:4])>=2022 else False`
   After the fix, Run3 correctly uses the reduced regrouped scheme
   (CMS_scale_j_Absolute, CMS_scale_j_FlavorQCD, CMS_scale_j_BBEC1,
   CMS_scale_j_HF, CMS_scale_j_EC2, CMS_scale_j_RelativeBal — fully
   correlated across eras — plus CMS_scale_j_Absolute_<era>,
   CMS_scale_j_BBEC1_<era>, CMS_scale_j_HF_<era>,
   CMS_scale_j_EC2_<era>, CMS_scale_j_RelativeSample_<era> — decorrelated
   per era, using the ERA_MAP: 2022preEE->2022, 2022postEE->2022EE,
   2023preBPix->2023, 2023postBPix->2023BPix, 2024->2024, 2025->2025,
   2026->2026), matching the standard JERC-recommended reduced set
   used for Run3 CMS analyses to keep the JES nuisance-parameter count
   manageable. Not committed in the nested tau repo — see COMMIT
   STATUS below; edit is in place on disk.

======================================================================
CHECKED AND FOUND CORRECT
======================================================================

- JEC application order/method (src/jets.cxx: RawPt -> PtCorrectionL1
  [L1FastJet] -> PtCorrectionL2L3 [L2Relative, then L2L3Residual for
  data only] -> JER hybrid smearing for MC): matches the "Mandatory
  Jet Energy Corrections at CMS" table in
  More_on_Jet_Energy_Corrections doc — MC: L1 + MC-truth(L2Relative);
  Data: L1 + MC-truth(L2Relative) + L2L3Residuals. The doc's note
  that "PUPPI jets do not need the L1 Pileup corrections... a dummy
  file is provided to maintain compatibility with Run2 scripts" means
  applying L1FastJet unconditionally for Run3 (as the code does) is
  correct/harmless — the correction JSON itself is a no-op for PUPPI.

- JER hybrid smearing (src/jets.cxx PtCorrectionL2L3, ~lines 400-463):
  scaling method when a gen-matched jet is found (ΔR<Rcone/2 and
  |pT-pT_ptcl|<3σ_JER·pT), else stochastic smearing
  `c_JER = 1 + N(0,σ)·sqrt(max(s²-1,0))`, truncated at 0 via
  `std::max(0.0, 1.0+jer_corr)`. This matches the Jet_Energy_Resolution
  doc's formulas and its recommendation to use the "hybrid method"
  exactly, including the ΔR and 3σ matching cuts and the truncation
  rule.

- Era-dependent evaluator signatures (L2Relative gains a phi argument,
  L2L3Residual gains a run argument, for era>=2023postBPix / data)
  are handled consistently in PtCorrectionL2L3, PtCorrectionMC, and
  PtCorrectionData.

- Jet veto map (src/jets.cxx VetoMap(), producers/jets.py
  JetVetoMapVeto, config.py jet_veto_* parameters): applied with
  min_pt=15 GeV, id_wp=6 (tightLepVeto), max_em_frac=0.9
  (chEmEF+neEmEF<0.9), matching the standard JMAR/JERC jet-veto-map
  selection recipe; evaluator inputs are clamped to the map's valid
  eta/phi range before evaluation, matching typical correctionlib
  usage for these maps. Applied identically to data and MC for all
  Run3 eras (mandatory per the Jet_Veto_Map doc and the VetoMap()
  docstring's own citation of
  https://cms-jerc.web.cern.ch/Recommendations/#jet-veto-maps).
  Correctly removed only for Run2 (era<2022, where it is optional)
  via a modification_rule scoped to `if int(era[:4]) < 2022`; no such
  removal exists for Run3, so I initially suspected (then ruled out)
  that JetVetoMapVeto was being dropped for all "non-fake_era" samples
  — that removal rule only fires inside the Run2 branch.

- Veto-map file/name per era (config.py jet_veto_map_file,
  jet_veto_map_name): all 7 Run3 eras present, names match the known
  official jetvetomaps.json.gz keys (Summer22_23Sep2023_RunCD_V1,
  Summer22EE_23Sep2023_RunEFG_V1, Summer23Prompt23_RunC_V1,
  Summer23BPixPrompt23_RunD_V1, Summer24Prompt24_RunBCDEFGHI_V1,
  Summer24Prompt25_RunCDEFG_V1, Summer24Prompt26_RunBCD_V1).

- Missing-era check: all of 2022preEE, 2022postEE, 2023preBPix,
  2023postBPix, 2024, 2025, 2026 have complete jet_jec_file,
  jet_jes_tag, jet_jer_tag, jet_jec_algo (AK4PFPuppi throughout Run3),
  jet_veto_map_file/name entries in config.py, and are present in
  jet_variations.py's ERA_MAP and generate.py's available_eras. No
  era is silently missing JEC/JER/veto-map configuration.

- Reduced JES source list content (after fix #3): Absolute,
  FlavorQCD, BBEC1, HF, EC2, RelativeBal (correlated) +
  Absolute/BBEC1/HF/EC2/RelativeSample decorrelated by era — this is
  the standard JERC-recommended reduced/regrouped source grouping
  (matching the Uncertainty_Sources doc's five broad categories:
  Pileup, Relative eta-dependent, Absolute pT-dependent, Time
  Dependence, Flavor Response) and correctly maps decorrelation to the
  categories the doc explicitly calls out as being determined
  per-data-taking-period (time dependence, relative eta-dependent JER
  closure).

- JER uncertainty (jet_variations.py: `CMS_res_j_{era_tag}`) is
  decorrelated per era, consistent with each era having its own
  JER scale-factor/resolution JSON tag in config.py, and with the
  Uncertainty_Sources doc's explicit recommendation to treat JER and
  JEC/RelativeJER as uncorrelated (they are already separate,
  independently-named shifts in this code, so nothing to change).

- HEM 15/16 issue (2018-only) is correctly scoped to era=="2018" only.
