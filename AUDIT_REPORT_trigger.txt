WHAT WAS CHECKED
-----------------
1. include/triggers.hxx / src/triggers.cxx: matchParticle() bit-matching
   logic (trigger::matchParticle, ~line 90). Confirmed it requires ALL bits
   in the passed vector to be set (AND, not OR) on the TrigObj_filterBits
   bitmask, and pt_threshold=-1/etacut/deltaR logic looked correct. No
   changes needed — this is the correct semantic for NanoAOD's
   multi-filter-bit trigger objects (e.g. a PNet WP bit + common PNet bit +
   CrossL1 bit must all be set simultaneously).

2. producers/triggers.py: EVP wiring for SingleObjectFlag/DoubleObjectFlag
   producers per scope/vec_config. Wiring is consistent with
   tau_triggersetup.py's vec_config names (singletau_trigger_leading/
   trailing, doubletau_trigger, mutau_cross_trigger, eltau_cross_trigger,
   etc.). Minor code smell noted but NOT fixed (out of scope /
   low-risk/cosmetic): lines 39-46 define
   `with defaults(input=[q.p4_2] + TrigObj_collection):` twice in a row,
   redefining EMGenerateSingleMuonTriggerFlags identically before defining
   GenerateSingleTrailingTauTriggerFlags — harmless (both blocks are
   byte-identical) but redundant.

3. tau_triggersetup.py (2574 lines): reviewed every EraModifier block for
   singlemuon_trigger, mutau_cross_trigger, singleelectron_trigger,
   eltau_cross_trigger, doubletau_trigger, elmu_cross_trigger,
   singletau_trigger_leading/trailing, doubleelectron_trigger, and the
   "TRIGGER SCALE FACTORS" Run-3 block (SF json paths, per-leg SF wiring,
   and the "trigger shifts" systematics block) for all 7 target eras.

4. producers/scalefactors.py: SingleEleTriggerSF, SingleMuTriggerSF,
   MuTauTriggerLeg1/2SF, EleTauTriggerLeg1/2SF, DoubleTauTriggerLeg1/2SF
   producers. Confirmed the per-leg efficiency method is used correctly
   for cross triggers (leg1 = lepton-leg SF from EGM/MUO POG jsons, leg2 =
   tau-leg SF from the TAU POG "tau_trigger" correction in tau.json.gz),
   matching the TWiki's statement that "SFs are only calculated for the
   tau leg of DiTau, DiTau+Jet, ETau, MuTau, and SingleTau" (the lepton-leg
   HLT SF is a separate, standard single-lepton trigger SF).

5. src/taus.cxx physicsobject::tau::scalefactor::Trigger(): confirmed via
   docstring/implementation that the correctionlib "tau_trigger" correction
   only exposes "nom"/"up"/"down" per corr_type (sf/eff_data/eff_mc) — i.e.
   CROWN produces one combined up/down shift per leg, and the TWiki's
   TauPOG systematics naming (systematics_master.txt: CMS_trig_t_<ditau|
   etau|mutau|...>_<WP>_eff_data_<era> / eff_mc_<era>, decorrelated per
   era) is realized by the downstream datacard/combine step renaming that
   single CROWN shift into per-era nuisance names, not by CROWN itself
   producing multiple variation strings. This is standard practice and NOT
   a bug.

6. "ditau_trigger_wp": "Medium" / "ditau_trigger_corrtype": "sf"
   (config.py:854-855) applied uniformly across eras regardless of whether
   the era's HLT uses DeepTau or PNet tagging — this matches how the TAU
   POG correctionlib json is structured (WP name is tagger-agnostic; the
   json itself differs per era file) and is consistent with the doubletau
   trigger's own WP choice ("Medium", matching HLT_Double*_Medium_* in all
   eras). No change made.

ISSUES FOUND AND FIXED (all in analysis_configurations/tau/tau_triggersetup.py)
---------------------------------------------------------------------------

A. mutau_cross_trigger and eltau_cross_trigger used stale DeepTau HPS HLT
   path names for 2025 and 2026.
   - Doc: TauTrigger_CMS_TWiki.txt line 211 ("Trigger Table for 2025"):
     "In 2025, DeepTau triggers were set to PS=0, meaning they will remain
     in the menu for technical reasons but no data will be collected with
     them." (CMSHLT-3447/CMSHLT-3240). Line 515 (Updates for 2025):
     "DeepTau removed from GRun (previously only deactivated by PS=0)"
     (CMSHLT-3567). The 2025 trigger table (lines 285-335) lists only PNet
     paths for MuTau/ETau, e.g. HLT_IsoMu20_eta2p1_PNetTauhPFJet27_Medium_
     eta2p3_CrossL1 and HLT_Ele24_eta2p1_WPTight_Gsf_PNetTauhPFJet30_
     Medium_eta2p3_CrossL1.
   - Code before: for "2025" and "2026", mutau_cross_trigger used
     hlt_path "HLT_IsoMu20_eta2p1_LooseDeepTauPFTauHPS27_eta2p1_CrossL1"
     (flagname "trg_cross_mu20tau27_hps"), and eltau_cross_trigger used
     "HLT_Ele24_eta2p1_WPTight_Gsf_LooseDeepTauPFTauHPS30_eta2p1_CrossL1"
     (flagname "trg_cross_ele24tau30_hps") — both identical copies of the
     2022-2024 entries. Since these HLT paths deliver essentially zero
     data after PS=0 (and are gone entirely later in 2025), the trigger
     flag would almost never be true in real 2025/2026 data.
   - Fix: for 2025 and 2026, switched to the PNet Medium-WP path
     (matching the WP already used for doubletau_trigger's 2024-2026
     entries, "trg_double_tau30_mediumiso_pnet" ==
     HLT_DoublePNetTauhPFJet30_Medium_L2NN_eta2p3):
       mutau: "HLT_IsoMu20_eta2p1_PNetTauhPFJet27_Medium_eta2p3_CrossL1",
              p2_filterbit "1, 4, 13" (WP-Medium bit 1, PNet common bit 4,
              CrossL1/MuTau bit 13, per the doc's V14/V15 NanoAOD
              triggerbit column for that path)
       eltau: "HLT_Ele24_eta2p1_WPTight_Gsf_PNetTauhPFJet30_Medium_eta2p3_
              CrossL1", p2_filterbit "1, 4, 12"
     2024 was left untouched (DeepTau HPS): the doc explicitly states
     "In 2024, both DeepTau and PNet triggers were available for the
     entire duration of data-taking. No recommendation has been made by
     Tau POG on which trigger to use" — so the existing 2024 choice is a
     valid option, not a bug, even though it differs from the
     already-PNet doubletau_trigger choice for 2024 (also explicitly
     doc-sanctioned since "no recommendation" applies per-trigger).
   - Cascading fix: the trigger-SF leg1/leg2 lookups for mutau/eltau
     (both the nominal "mutau_trigger_leg1_sf"/"mutau_trigger_leg2_sf"/
     "eletau_cross_trigger_leg1_sf"/"eletau_cross_trigger_leg2_sf" blocks,
     and the MuTauTriggerSF{UP,DOWN}/EleTauTriggerSF{UP,DOWN} shift
     blocks) hardcoded the flag name "trg_cross_mu20tau27_hps" /
     "trg_cross_ele24tau30_hps" as a plain string for ALL eras. Since the
     produced flagname now differs by era, these lookups were wrapped in
     EraModifier (mirroring the pattern already used for
     doubletau_trigger_leg1/leg2's "doubletau_trigger_flag", which
     correctly switches "trg_double_tau35_mediumiso_hps" ->
     "trg_double_tau30_mediumiso_pnet" at the 2023BPix/2024 boundary).
     Without this fix, the rename in item A alone would have broken tau-
     leg and lepton-leg trigger SF application for 2025/2026 (the SF
     producer would look up a branch, "trg_cross_mu20tau27_hps", that no
     longer exists for those eras).

B. singletau_trigger_leading / singletau_trigger_trailing used a
   Run-2-era HLT path name for ALL Run3 eras (2022preEE..2026), explicitly
   marked in-code "## TODO: not implemented, here as a placeholder".
   - Code before: hlt_path
     "HLT_MediumChargedIsoPFTau180HighPtRelaxedIso_Trk50_eta2p1" (a
     pre-DeepTau, Run2 ChargedIso path name) copy-pasted unchanged into
     every 2022preEE/2022postEE/2023preBPix/2023postBPix/2024/2025/2026
     era entry. This path does not exist in the Run3 HLT menu (Run3
     SingleTau uses DeepTau/PNet tagging, e.g.
     HLT_LooseDeepTauPFTauHPS180_L2NN_eta2p1); the flag would essentially
     never fire for Run3 data/MC.
   - Doc: 2023 table (line 1057) and 2024 table list SingleTau path
     "HLT_LooseDeepTauPFTauHPS180_L2NN_eta2p1", NanoAOD V14 triggerbit
     "3, 9". The 2025 table (lines 225-247) lists only the PNet path
     "HLT_SinglePNetTauhPFJet130_Medium_L2NN_eta2p3" (triggerbit "1, 4, 9"
     for Medium WP), consistent with DeepTau being PS=0/removed in 2025.
   - Fix: replaced the placeholder for 2022preEE/2022postEE/2023preBPix/
     2023postBPix/2024 with HLT_LooseDeepTauPFTauHPS180_L2NN_eta2p1,
     ptcut 180, filterbit "3, 9"; and for 2025/2026 with
     HLT_SinglePNetTauhPFJet130_Medium_L2NN_eta2p3, ptcut 130, filterbit
     "1, 4, 9". Removed the stale "TODO: not implemented" comments for
     these eras and added a comment explaining the era split. 2017/2018
     entries (which use the same Run2 path name but for which that name
     IS the genuine 2017/2018 HLT path) were left untouched.
   - No downstream SF or config.py selection currently references
     trg_single_tau180_1/2 / trg_single_tau130_pnet_1/2 by name (grepped
     analysis_configurations/tau for all .py files) — the TWiki notes
     SingleTau trigger SFs are derived separately and are not wired into
     this SF section at all (for any era, including Run2), so this fix is
     restricted to the trigger-matching flag itself; no SF wiring exists
     to update. This is a pre-existing (undocumented-as-gap) limitation,
     not something introduced or fixed here.

GAPS / COULD NOT FULLY VERIFY
-------------------------------
- mutau_cross_trigger's tau-leg filterbit for the 2022-2024 DeepTau HPS
  path is set to a single bit ("p2_filterbit": 9), whereas the TWiki's
  V14/V15 NanoAOD triggerbit column for
  HLT_IsoMu20_eta2p1_LooseDeepTauPFTauHPS27_eta2p1_CrossL1 lists "3, 13"
  (two bits) for the tau object. This is a potential discrepancy, but I
  did NOT change it: this single-bit convention is used identically for
  eltau (p2_filterbit=3) and is long-standing in the file (predates this
  audit, used across all Run2 and Run3 DeepTau-era entries), so it may
  reflect an intentional/validated CROWN-specific bit convention (NanoAOD
  filter-bit indices are not guaranteed to be identical across production
  campaigns/versions even for "the same" trigger object) that I could not
  confirm without deeper access to the actual NanoAOD filterBits
  definition used by this analysis's production campaign. Given this
  numbering is used across an already-running analysis (branch
  "sm_htt_run3"), changing it without being certain risked a much larger
  regression than the two fixes made above, both of which had a clean,
  already-established in-file precedent (doubletau_trigger's PNet
  migration) to model from. Recommend a follow-up check against the
  actual NanoAOD Tau_filterBits/TrigObj_filterBits documentation for the
  specific production campaign(s) in data/hleprare and
  data/jsonpog-integration, or by inspecting a NanoAODv15 file directly.
- Could not fetch any of the ~112 links in TauTrigger_CMS_TWiki_links.txt
  (gitlab.cern.ch SF measurement repos, indico) — no network/proxy check
  was attempted in the time available for this pass; the correctionlib
  json paths referenced in tau_triggersetup.py (data/hleprare/
  TriggerScaleFactors/.../CrossMuTauHlt_MuLeg_v1.json,
  CrossEleTauHlt_EleLeg_v1.json, and the tau.json.gz TAU POG file) were
  taken at face value from config.py/tau_triggersetup.py; their internal
  correction-name contents (e.g. "mutau"/"etau"/"ditau" trigger_name
  keys) were not independently verified against a live json file.
- 2024/2025/2026 mu-leg and ele-leg cross-trigger SF json files
  (mutau_cross_trigger_leg1_sf_file, eletau_cross_trigger_leg1_sf_file,
  singleelectron_trigger_sf_file) all reuse the 2023postBPix / 2024
  json as a placeholder for 2025/2026 (no dedicated 2025/2026 measurement
  published yet per the TWiki, which itself has no "Trigger SFs in Run3"
  entries beyond 2022/2023/2024 SingleTau and PNet/DeepTau 2024 SFs).
  This is a genuine doc gap for those eras (item #6 in the task) but not
  a code bug — flagged, not changed, since there is nothing more correct
  to substitute yet.
- tau_embedding_settings.py (not in the originally-listed file set) also
  hardcodes "trg_cross_mu20tau27_hps" for eras including 2025/2026;
  since this is the embedding-sample trigger config (a distinct code path
  from tau_triggersetup.py, used only for embedded-tau-pair replacement
  samples), it was left untouched — out of the stated scope
  (producers/triggers.py, config.py, tau_triggersetup.py,
  producers/scalefactors.py) and embedding trigger requirements may
  legitimately need to track the original recorded trigger rather than
  the current-menu trigger. Flagged for awareness only.
