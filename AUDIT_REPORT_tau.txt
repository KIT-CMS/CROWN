AUDIT REPORT: Run3 htautau tau ID / energy scale / scale-factor bookkeeping
============================================================================
Scope: TauPOG DeepTau2018v2p5 ID, tau energy scale (ES) corrections, and
ID/ES scale-factor (SF) systematics for eras 2022preEE..2026.

IMPORTANT NOTE ON REPO LAYOUT
------------------------------
The tau analysis config (config.py, producers/taus.py, producers/scalefactors.py)
does NOT live in the CROWN repo itself. It lives in a separate git repo,

worktree + branch for it, matching the pattern already used by the parallel
btag/zpt audits:

  Run 2 and Run 3 through 2026, including the newer 2024/2025 pT-binned TES
  measurement scheme.
- CMS TWiki (lower precedence): TauIDRecommendationForRun3, last edited
  2026-01-13 by AlexeiRaspereza -- but its content is explicitly scoped to

- Ground truth verification: the actual deployed correctionlib jsons on

CONFLICT RESOLUTION (CMS TWiki vs TauPOG Wiki)
------------------------------------------------
1. Available VSjet working points for 2022/2023: the CMS TWiki claims
   "VSjet = Loose, Medium, Tight, VTight" are available. The TauPOG Wiki
   states only Medium is measured/available for 2022, 2022EE, 2023, 2023BPix
   ("VsJet : M"). I verified the deployed json (2022preEE) and it only has a
   'wp' category with a single key 'Medium' for the DM-binned SF -- so the
   TauPOG Wiki is correct and the TWiki is stale here. The code already only
   enables the Medium WP for these eras (config.py, `vsjet_tau_id` dict,
   comment "for Run 3 new TAU corrections, only the Medium wp sf are
   provided") -- this was already correct, no change needed. Documenting the
   conflict per instructions.
2. For 2024/2025 the TauPOG Wiki states VSjet Loose..VTight WPs are now
   available (confirmed also in the 2024 json: 'wp' category has
   ['Loose','Medium','Tight','VTight']), but the code still only uses Medium
   for all Run3 eras uniformly. This is a legitimate analysis choice (single
   operating point), not a bug -- not changed, but noted as an opportunity if
   the analysis ever wants to add categories in other WPs.

WHAT WAS CHECKED AND FOUND CORRECT (no changes needed)
---------------------------------------------------------
- DeepTau2018v2p5 chosen as the tau ID algorithm for era >= 2022 (config.py
  "tau_id_algorithm": "DeepTau2018v2p5"), matching both docs' recommendation.
  DeepTau2017v2p1 correctly kept for era < 2022 (Run 2).
- Kinematic tau selection: pT >= 20 GeV, |eta| <= 2.5 (2.3 for tt, per
  analysis choice), |dz| <= 0.2 -- matches TauPOG Wiki recommendation exactly
  (analysis_configurations/tau/config.py, "min_tau_pt"/"max_tau_eta"/
  "max_tau_dz" parameters).
- Decay modes selected: 0, 1, 10, 11 ("tau_dms": "0,1,10,11") -- matches
  recommendation (DM 2/5/6 correctly excluded).
- Tau ES correction is applied per decay mode, with correct correction
  source json name "tau_energy_scale" (config.py "tau_ES_json_name").
  Verified against the deployed json: the correction is genuinely
  DM-and-pT-binned as documented, with the "flag=dm" node keyed by
  dm in {0,1,10,11} and (for 2022/2023) a further "wp"/"wp_VSe" split; for
  2024+ additionally binned in pT with edges [20,40,60,200].
- Tau ES uncertainty split by decay mode in the systematic variation naming:
  confirmed via src/taus.cxx PtCorrectionMC/PtCorrectionMC_v12 and
  producers/taus.py -- each DM has its own independent "tau_es_DM{0,1,10,11}"
  config key, wired to distinct shift names (tauEsDM0..tauEsDM11) in
  config.py. For era >= 2024, additionally split by pT bin
  (tau_es_DM{dm}_pt{20to40,40to60,60toInf}), matching the TauPOG Wiki's
  description of the 2024/2025 recipe ("TES up/down variations need to be
  decorrelated among DM bins and Eras, not per Tau-pT bin as one single TES
  is freely floated" -- i.e. one TES nuisance covers all 3 pT bins together
  per DM/era, which is exactly what the code's per-(DM,pT-bin) shift wiring,
  each independently perturbing only its own bin's correction-string
  argument while leaving the fit function otherwise nominal, achieves).
  No merging into a single inclusive ES shift was found -- good.
- Electron-fake tau ES/SF barrel/endcap boundary: hardcoded at |eta| = 1.5 in
  src/taus.cxx (PtCorrectionMC_eleFake, `barrel_end_cut = 1.5`), matching the
  TauPOG Wiki's "two bins for barrel and endcap, separated at 1.5" exactly.
- Muon-fake tau ID SF eta "wheel" binning: src/taus.cxx uses edges
  {0.0, 0.4, 0.8, 1.2, 1.7, [2.5]}, matching the TauPOG Wiki's
  "[0.0, 0.4, 0.8, 1.2, 1.7, 2.4]" binning (code extends the last bin to 2.5
  to cover the full muon-system acceptance / tau eta cut, a reasonable and
  harmless choice since no tau above 2.5 survives the kinematic selection).
- Muon-fake and electron-fake tau ID SF/ES: verified via json inspection
  that "DeepTau2018v2p5VSe"/"VSmu" and "tau_energy_scale" (for genmatch
  1/2/3/4/6) all cleanly expose 'nom'/'up'/'down' syst keys for every era
  checked -- the simple nom/up/down variation strings used by the code are
  therefore correct for these corrections (unlike vsJet ID SF, see Issue #2).
- vsJet/vsEle/vsMu WP combinations used for ID SF computation (Medium vsJet,
  VVLoose/Tight vsEle, VLoose/Loose/Medium/Tight vsMu combined with
  VVLoose/Tight vsEle) match the TauPOG Wiki's documented combination table
  for 2022-2025.
- `tau_vsjet_sf_dependence = "dm"` is used uniformly, matching the
  recommendation that DM-binned (fit-based, continuous-in-pT) SFs be used
  "in all cases" rather than a separate high-pT flag path; the code does not
  attempt to use the (WIP / partially-implemented per TauPOG) dedicated
  high-pT SF branch, which is consistent with the recommended default.

ISSUES FOUND AND FIXED
------------------------

Issue #1 -- typo breaking 2024+ vsJet ID SF systematic shifts (config.py)
  Before: `shift_key=f"tau_od_vsjet_DM{dm}_pt{pt}"` (line ~2584).
  The declared config parameters are named "tau_id_vsjet_DM{dm}_pt{pt}"
  (config.py lines ~812-823); "tau_od_vsjet_..." does not exist anywhere
  else in the file. This silently broke the Up/Down wiring for the vsJet
  ID SF systematic for all DM/pT bins in 2024, 2025 and 2026.
  Fix: corrected to `shift_key=f"tau_id_vsjet_DM{dm}_pt{pt}"`.
  Commit: 5c8f8ed.

Issue #2 (most significant) -- vsJet ID SF Up/Down systematic silently a
no-op for 2022preEE/2022postEE/2023preBPix/2023postBPix (config.py)
  The code (add_shift for name=f"vsJetDM{dm}", shift_key=f"tau_id_vsjet_DM{dm}",
  shift_map={"Up": "up", "Down": "down"}) passed the literal strings
  "nom"/"up"/"down" as the correctionlib "syst" argument for the DM-binned
  DeepTau2018v2p5VSjet SF (`Tau_1/2_VsJetTauID_SF_v12` producers).
  I inspected the actual deployed json for all four pre-2024 Run3 eras
  (/cvmfs/cms-griddata.cern.ch/.../TAU/.../tau.json.gz) and found the "syst"
  category for the DM-binned ("flag=dm") vsJet correction does NOT contain
  plain "nom"/"up"/"down" keys. It only contains the granular, individually
  named components documented on the TauPOG Wiki's "Run 3 (2022-2023) - same
  as Run 2" recipe:
    stat1_dm<DM>_{up,down}, stat2_dm<DM>_{up,down},
    syst_alleras_{up,down}, syst_<era>_{up,down} (e.g. syst_2022_preEE_up),
    syst_TES_<era>_dm<DM>_{up,down}
  and a `"default"` fallback formula (the nominal SF fit) used for *any*
  string that does not match one of those exact keys. Since "nom", "up" and
  "down" never match any of the 10 explicit keys, ALL THREE fell through to
  the same nominal-formula default -- meaning the Up and Down systematic
  variations for the tau ID vsJet SF were completely inert (identical to
  nominal) for 2022preEE, 2022postEE, 2023preBPix and 2023postBPix. This is
  a real, previously undetected bug: the corresponding shape-uncertainty
  templates in any datacard/fit using this Ntuple production would have been
  bit-identical to nominal, understating this SF uncertainty entirely for
  four of the seven Run3 eras.
  (For comparison, I verified the 2024 json's vsJet SF "syst" category DOES
  contain clean 'nom'/'up'/'down' keys per (DM, pT-bin), so era >= 2024 was
  NOT affected by this issue -- only the pre-2024 DM-only scheme was broken,
  and only after fixing Issue #1 as a prerequisite for the 2024+ path.)

  Fix: replaced the single generic "up"/"down" shift per DM with five
  correctly-named, individually keyed shifts per DM (for era < 2024), each
  passing the real json key:
    - vsJetStat1{era}DM{dm} -> stat1_dm{dm}_{up,down}   (decorrelated per DM+era)
    - vsJetStat2{era}DM{dm} -> stat2_dm{dm}_{up,down}   (decorrelated per DM+era)
    - vsJetSystTES{era}DM{dm} -> syst_TES_{era_key}_dm{dm}_{up,down}
                                                          (decorrelated per DM+era)
    - vsJetSystAllEras -> syst_alleras_{up,down} applied simultaneously to all
      4 DM parameters (fully correlated across DM and, since the same shift
      *name* "vsJetSystAllEras" -- with no era-dependent suffix -- is emitted
      identically by every era's config, correlated across eras too, per the
      TauPOG Wiki: "A systematic uncertainty fully correlated between eras
      and DMs").
    - vsJetSyst{era} -> syst_{era_key}_{up,down} applied to all 4 DM
      parameters simultaneously (correlated across DM, and -- since the
      shift name embeds `era` -- automatically decorrelated across eras when
      per-era Ntuples are combined downstream, per "Systematic uncertainties
      correlated between DMs, but uncorrelated by eras").
  This exactly reproduces the uncertainty-component breakdown and
  correlation scheme documented for "Run 3: 2022_preEE, 2022_postEE,
  2023_preBPix, 2023_postBPix -- Same as Run 2" in both the TauPOG Wiki and
  (for the component list, though not the era coverage) the CMS TWiki.
  Verified: `python3 -c "import ast; ast.parse(...)"` succeeds, and a live
  `build_config("2022preEE", ...)` call was run through the full producer/
  shift wiring in `code_generation` (reaching the final `configuration.
  validate()` step -- which then hits an unrelated, pre-existing
  dictionary-mutation bug in code_generation/configuration.py, reproduced
  identically on the unmodified pre-fix config, i.e. NOT caused by this
  change and out of scope for this audit) with no exceptions raised by the
  new shift-wiring code itself.
  Commit: 5c8f8ed.

Issue #3 -- undocumented placeholder reuse of 2024/2025 json for 2026
  (config.py "tau_sf_file", era "2026")
  The 2026 entry silently reused the exact same json path as 2024/2025 with
  no comment. Per the TauPOG Wiki status table (2026-06-10), ALL 2026 tau
  corrections (genuine tau ID SF, ES, e/mu fake SF/ES, trigger SF) are
  marked "Not available yet" (all red X). Continuing to use the 2024/2025
  json as a stand-in is a reasonable stopgap (consistent with how several
  other correction types in this config already handle 2026 placeholders),
  but it was undocumented and easy to mistake for an official 2026
  measurement. Also documented that the 2025 reuse (same json as 2024) IS
  the official TauPOG recommendation (same MC campaign), unlike 2026.
  Fix: added explanatory comments in config.py distinguishing the two cases.
  Commit: c68c8f4. No file path change made since no real 2026 json exists
  yet to substitute.

CORRELATION ACROSS ERAS (item 5 of the task) -- COULD NOT FULLY VERIFY
--------------------------------------------------------------------------
CROWN's `add_shift`/`SystematicShift` mechanism names each systematic
identically regardless of which era's config.py invocation produced it,
*except* where the config author chooses to embed `era` into the shift
`name` string (which nothing in this file did prior to my Issue #2 fix,
across ALL producer domains -- jets, met, muons, tau, etc. -- confirmed by
grepping the whole file for `name=f"..."` patterns including `{era}`).
Whether two identically-named systematics from different eras' Ntuples end
up correlated or decorrelated in a final fit is therefore decided entirely
downstream, outside CROWN (in whatever datacard/harvesting tool combines the
per-era friend trees) -- this repository does not have visibility into that
step, so I could not verify end-to-end whether the intended TauPOG
correlation scheme (e.g. "uncorrelated across years" for e/mu-fake SF, or
the syst_alleras/syst_<era> split for vsJet SF) is actually honored in the
final statistical analysis. I *did* fix the vsJet SF's era-decorrelation at
the CROWN level (Issue #2, by embedding `era` in the relevant shift names)
since that is a case where the JSON's uncertainty structure and the naming
scheme genuinely differ per era and there was no way to get a well-defined
answer without doing so. For all other tau-related systematics (e/mu-fake ID
SF and ES, which the TauPOG Wiki says must be uncorrelated across years),
the shift names do NOT embed era, consistent with the rest of the codebase's
convention of leaving era-(de)correlation to the downstream combination step
-- flagging this as a "not independently verifiable from CROWN alone" item
rather than changing it, since changing only the tau-specific shifts while
leaving every other systematic in the framework using the opposite
convention would be an inconsistent, likely-wrong intervention without
visibility into how the downstream tool actually interprets shift names.

7-ERA CORRECTION FILE WIRING COMPLETENESS (item 6)
------------------------------------------------------
All 7 Run3 eras (2022preEE, 2022postEE, 2023preBPix, 2023postBPix, 2024,
2025, 2026) have a `tau_sf_file` entry (no "TO_ADD" placeholders, unlike
some Run2 btag entries in the same file). 2022-2024 point at
era-specific, TauPOG-measured jsons; 2025 correctly reuses the 2024
json per the TauPOG Wiki's explicit recommendation; 2026 reuses the same
json but has NO official measurement behind it yet (see Issue #3) --
this is the one genuine "gap for a later era" per item 6 of the task, and
it is inherent to the current state of TauPOG's Run3 program (not fixable
by this repository).
